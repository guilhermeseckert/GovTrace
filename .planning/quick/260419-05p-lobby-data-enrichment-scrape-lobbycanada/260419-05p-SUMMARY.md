# Phase quick Plan 260419-05p: Lobby Data Enrichment from Secondary CSVs

**One-liner:** ZIP-extraction and in-memory Map join of secondary lobbycanada.gc.ca CSVs to populate subject_matter, target_departments, public_official_name, public_official_title, and department fields that were previously null/"Unknown".

## What Was Built

### Downloader Changes (both registrations and communications)

Both `downloadLobbyRegistrations` and `downloadLobbyCommunications` now extract ALL `.csv` files from the ZIP instead of only `PrimaryExport`. The `DownloadResult` interface gains an `extractedFiles: Record<string, string>` field mapping filename to absolute path. The `localPath` field still points to PrimaryExport for backward compatibility.

### Registration Parser Enrichment

`parseLobbyRegistrationsFile` now accepts an optional `extractedFiles` parameter. When provided:

1. `Codes_SubjectMatterTypesExport.csv` (54 rows) parsed into a code→English name Map
2. `Registration_SubjectMattersExport.csv` (~997K rows) parsed into a `Map<reg_id_enr, string[]>` of resolved SMT names
3. `Registration_GovernmentInstExport.csv` (~1.78M rows) parsed into a `Map<reg_id_enr, string[]>` of English institution names
4. During main parse loop: `reg_id_enr` extracted from each primary row as join key; subject matters and departments looked up from Maps; falls back to primary CSV columns if no secondary data found

**Critical implementation detail:** The join key is `reg_id_enr` (e.g., `611057`), not `reg_num_enr` (e.g., `777408-4993-11`). Secondary files only carry `reg_id_enr`.

### Communications Parser Enrichment

`parseLobbyCommunicationsFile` now accepts an optional `extractedFiles` parameter. When provided:

1. `Codes_SubjectMatterTypesExport.csv` parsed into code→English name Map
2. `Communication_SubjectMattersExport.csv` (~649K rows) parsed into `Map<comlog_id, string[]>`
3. `Communication_DpohExport.csv` (~549K rows) parsed into `Map<comlog_id, DpohRecord[]>` where each record has name, title, department, branch
4. During main parse loop: first DPOH used for flat fields (`publicOfficialName`, `publicOfficialTitle`, `department`); all DPOHs stored in `rawData.dpoh_officials`; subject matters joined with `"; "` separator

**ID preservation:** The `comlog_id`-based `deriveSourceKey([comlogId])` ID derivation is unchanged, ensuring enriched records update existing rows rather than creating duplicates.

## Deviations from Plan

None — plan executed exactly as written. All pitfalls from RESEARCH.md were handled:
- Pitfall 1 (reg_id_enr vs reg_num_enr): join uses `row.reg_id_enr` explicitly
- Pitfall 2 (Latin-1 encoding): `detectAndTranscode` applied to all secondary CSVs
- Pitfall 3 ("null" string): `nullIfNullString()` helper filters `"null"` to `null`
- Pitfall 4 (array overwrite): Maps use `existing.push()` pattern, never overwrite
- Pitfall 5 (duplicate IDs): comlog_id-based ID derivation unchanged

## Known Stubs

None. All enrichment fields are fully wired. The data will be populated on next ingestion run.
