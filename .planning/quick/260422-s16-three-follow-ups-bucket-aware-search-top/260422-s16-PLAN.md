  <name>Task 1: fix(search): bucket-aware top-N so politicians surface</name>
  <files>apps/web/src/server-fns/search.ts</files>
  <action>
Modify `searchEntities` in `apps/web/src/server-fns/search.ts` ONLY. Do not touch `getAutocomplete`.

Goal: When the caller asks for everything (`data.type === 'all'`) and the query is substantive (`normalizedQuery.length >= 2`), guarantee that each of the three major entity buckets gets a fair share of top results by running THREE parallel SQL queries each limited to 10 rows, then merging. This prevents shorter names (e.g. "JUSTIN" the donor) from dominating pg_trgm similarity and burying longer politician names like "TRUDEAU, Justin".

Buckets:
- Politician: `entity_type = 'politician'`
- Org-group: `entity_type IN ('organization', 'company', 'department')`
- Person-group: `entity_type = 'person'`

Implementation notes:
1. Keep the existing shape of the SQL (pg_trgm `%` + ILIKE fallback, score = similarity, province/date filters applied identically).
2. Each bucket query uses `LIMIT 10` and NO offset (offset doesn't make sense across 3 merged buckets for a "top-N" view; pagination still works in the single-bucket branches below).
3. Use `Promise.all([...])` for the three queries. Merge results into a single `results` array; preserve per-bucket ordering by concatenating politician → org-group → person-group (so politicians surface first for name-type queries — this is the whole point).
4. Dedupe by `id` using a `Set` while merging, because overlapping entity_type would be rare but not impossible across historical data.
5. Cap the final merged list at 30 (i.e. 10 × 3) before the downstream `getEntityCounts` call, since `entityIds.length` drives per-row count queries.
6. Apply this ONLY when BOTH conditions hold: `data.type === 'all'` AND `normalizedQuery.length >= 2`. Otherwise fall through to the existing branches:
   - `data.type !== 'all'` and query >= 2 → single-bucket similarity path (unchanged)
   - query < 2 → browse mode sorted by canonical_name (unchanged)
7. Keep existing `provinceFilter`, `dateFromFilter`, `dateToFilter` applied to every bucket query. Do NOT apply `typeFilter` in the bucket branch (each bucket has its own hardcoded `entity_type` predicate).
8. Do NOT change the return shape or field names. `results.map(...)` logic at the bottom stays the same.
9. Do NOT introduce `any`. All rows must be typed as the existing anonymous `{ id; canonical_name; entity_type; province; score }` shape (extract a local type alias if it helps readability; do not export it).
10. Respect Ultracite/Biome: `for...of` over `.forEach`, nullish coalescing where applicable, no barrel files, no `console.log`.

Concrete SQL sketch (use Drizzle `sql` template; do not regress to raw strings):
```
-- politician bucket
SELECT id, canonical_name, entity_type, province,
       similarity(normalized_name, ${normalizedQuery}) AS score
FROM entities
WHERE (normalized_name % ${normalizedQuery} OR normalized_name ILIKE ${likePattern})
  AND is_active = true
  AND entity_type = 'politician'
  ${provinceFilter} ${dateFromFilter} ${dateToFilter}
ORDER BY score DESC
LIMIT 10

-- org-group bucket: entity_type IN ('organization', 'company', 'department')
-- person-group bucket: entity_type = 'person'
```
