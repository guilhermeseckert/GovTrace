# Phase 4.2: Entity Profile Storytelling - Research

**Researched:** 2026-04-02
**Domain:** React UI restructure, Claude prompt engineering, SQL pattern detection
**Confidence:** HIGH

## Summary

Phase 4.2 is a pure UI/UX restructure plus prompt upgrade — no new data sources, no schema migrations, no infra changes. All data needed for storytelling already exists in `entity_connections` (pre-computed pairwise relationships with `totalValue`, `transactionCount`, `sourceRecordIds`) and `entities` (canonical names). The plan must reshape how that data is presented, not how it is stored.

The work breaks into four cohesive areas: (1) upgrade `buildSummaryPrompt` in `summary.ts` to fetch and inject the top 20 connections as named context before generating the Claude summary; (2) build a new `ConnectionCards` component that renders each connection as a plain-English sentence reusing the `CONNECTION_LABELS` map already in `ConnectionsTable.tsx`; (3) add a new `getPatternCallouts` server function that runs SQL GROUP BY queries on `entity_connections` to detect four named patterns; and (4) restructure `entity/$id.tsx` so the default view is story-mode (AI story → callouts → cards → graph) with a toggle to reveal the full `ProfileTabs`.

The only true implementation complexity is the `getPatternCallouts` query logic and the enriched Claude prompt. Everything else is component wiring and JSX restructuring on top of a solid existing foundation.

**Primary recommendation:** Read `ConnectionsTable.tsx` lines 59–65 and 258–285 first — the color map and mobile card pattern are the direct building blocks for `ConnectionCards`, and reusing them exactly avoids visual inconsistency.

### Pattern 1: Enriched Summary Prompt (D-01 through D-04)

The current `buildSummaryPrompt` passes only counts. The upgrade queries `entity_connections` joined to `entities`, formats the top 20 as a structured list, and injects them into the prompt.

**Key query for enriched context:**
```typescript
// Source: existing getConnections pattern in datasets.ts (lines 251-291)
// Query both directions (entityA and entityB) then union and sort by totalValue desc
const topConnections = await db
  .select({
    connectedEntityName: entities.canonicalName,
    connectedEntityType: entities.entityType,
    connectionType: entityConnections.connectionType,
    totalValue: entityConnections.totalValue,
    transactionCount: entityConnections.transactionCount,
  })
  .from(entityConnections)
  .innerJoin(entities, eq(entities.id, entityConnections.entityBId))
  .where(eq(entityConnections.entityAId, data.entityId))
  .orderBy(desc(entityConnections.totalValue))
  .limit(20)
// Repeat for entityBId direction, then merge and re-sort client-side
```

**Prompt injection pattern:**
```typescript
const connectionLines = topConnections
  .map(c => `- ${c.connectedEntityName} (${c.connectionType.replaceAll('_', ' ')}): ${formatAmount(c.totalValue)}, ${c.transactionCount} transactions`)
  .join('\n')

const prompt = `...
Known connections (top by value):
${connectionLines}

Write in plain language a 9-year-old could follow. Use rounded numbers.
Name specific people and companies. Connect dots across datasets.
ALWAYS end with: "Connections shown do not imply wrongdoing."
Maximum 80 words.`
```

**Important:** The current `buildSummaryPrompt` caps at 300 `max_tokens`. With richer context and a longer prompt, bump to 400-500 max_tokens for the named-connection story format. This is within claude-haiku-4-5 constraints and does not require model escalation.

### Anti-Patterns to Avoid

- **Fetching connections twice:** `ConnectionCards` uses the EXISTING `getConnections` server function with `sortBy: 'totalValue', sortDir: 'desc'`, not a new query. Don't create a duplicate server function.
- **Rebuilding `CONNECTION_LABELS`:** The color map lives in `ConnectionsTable.tsx`. Extract it to a shared file (e.g., `src/lib/connection-labels.ts`) for import by both `ConnectionsTable` and `ConnectionCards` — do NOT duplicate it inline.
- **Blocking the loader with pattern callouts:** Same rule as AISummary — fetch `getPatternCallouts` client-side in a `useQuery` hook, not in the route loader. Pattern detection may require multiple DB queries and should not block page paint.
- **Animating `ProfileTabs` mounting:** The toggle controls conditional rendering (`{showDetailedRecords && <ProfileTabs />}`). Don't use CSS `display:hidden` for the tabs — unlike the visualizations sub-tab pattern (where D3 must stay mounted), ProfileTabs has no stateful D3 instances and is safe to unmount/remount.
- **Hardcoding disclaimer text:** The "Connections shown do not imply wrongdoing." disclaimer already lives in `en.ts` as `en.profile.disclaimer`. The `ConnectionCards` and `PatternCallouts` components must reference this key, not inline the string.

## Common Pitfalls

### Pitfall 1: Bidirectional entity_connections Query
**What goes wrong:** Querying only `WHERE entity_a_id = $id` misses connections where the entity is `entity_b_id`. An entity appears on either side of the relationship depending on which direction was computed.
**Why it happens:** The `entity_connections` table uses a directed edge model (A→B), but entities can appear as either A or B.
**How to avoid:** Always query both `entityAId` and `entityBId` using `or()` — or make two separate queries and merge, as `getConnections` already does (datasets.ts lines 251-291). The enriched summary query must also query both directions before sorting by totalValue.
**Warning signs:** Entity profile shows 0 connections when the connections table clearly has entries.

### Pitfall 2: Claude Token Budget for Enriched Prompt
**What goes wrong:** Top 20 connection lines × ~50 chars each = ~1000 chars of context added. With the existing system prompt, total input may approach the current `max_tokens: 300` output limit causing truncated summaries that cut off mid-sentence.
**Why it happens:** `max_tokens` controls output length, not input. But a richer prompt produces a longer response. The current 60-word cap in the prompt will need adjustment to 80 words to allow full cross-referenced sentences.
**How to avoid:** Bump `max_tokens` to 450-500 in the `client.messages.create` call. Update the prompt word limit instruction to 80 words. The aiSummaries schema `summaryText` is `text` with no length constraint — no migration needed.
**Warning signs:** Summaries end abruptly without the disclaimer sentence.

### Pitfall 3: Pattern Callouts That Fire on Empty Data
**What goes wrong:** Pattern C (temporal clustering) requires both lobbying and contracts connections. Entities with only donations (e.g., private citizens who donated once) have no `vendor_to_department` or `lobbyist_to_official` connections, so most pattern queries return zero rows. The `PatternCallouts` component must handle 0 callouts gracefully — returning null or rendering nothing.
**Why it happens:** Pattern detection runs for every entity regardless of their connection profile.
**How to avoid:** Each pattern query returns null when not triggered. Collect non-null results; if the array is empty, the component returns null (no section rendered). Don't render an empty "Did you know?" section.
**Warning signs:** "Did you know?" section renders with no cards inside.

### Pitfall 4: AISummary Cache Invalidation
**What goes wrong:** Cached summaries generated before this phase contain count-only content ("donated to 3 parties, received 2 contracts"). After D-01 lands, new summaries will be enriched with named connections. But the cache hit check in `getOrGenerateSummary` (`isStale = false`) will serve old summaries forever.
**Why it happens:** The `aiSummaries` table has no content-versioning field — just `isStale`. The existing weekly `isStale` flip (via ingestion pipeline) is the only invalidation mechanism.
**How to avoid:** Add a one-time bulk `UPDATE ai_summaries SET is_stale = true` migration step at deploy time, OR add a prompt version field and regenerate on version mismatch. Simplest approach: the plan should include a migration task that marks all existing summaries stale so they regenerate with the enriched prompt on next request.
**Warning signs:** Entity profiles continue showing old count-only summaries after deploy.

### Pitfall 5: Disclaimer Visibility in Story Mode
**What goes wrong:** The "Connections shown do not imply wrongdoing" disclaimer is currently inside `ProfileTabs` (as a tab-specific note on the connections and visualizations tabs). After the restructure, the primary story-mode view shows connections outside `ProfileTabs`. The disclaimer must remain visible in story mode — not only inside the now-collapsed tabs.
**Why it happens:** Disclaimer placement was designed for the tab context, not for top-level story mode.
**How to avoid:** Add `en.profile.disclaimer` to `ConnectionCards` (below the cards, matching current `AISummary` footer pattern) and/or to the story-mode section. The caveat in `en.profile.connections_disclaimer` and `en.profile.disclaimer` are identical — use either key.
**Warning signs:** Story mode shows connection cards without any legal caveat visible.

### Pitfall 6: ProfileTabs `connections` Count = 0
**What goes wrong:** `getEntityStats` currently returns `connections: 0` hardcoded (entity.ts line 178). The `ProfileTabs` tab bar shows badge counts — the Connections tab badge will always show 0, which looks broken especially when `ConnectionCards` above shows 10+ connections.
**Why it happens:** Connection count was never wired (hardcoded zero in `getEntityStats`).
**How to avoid:** Fix `getEntityStats` to count connections from `entityConnections` as part of this phase. Use `or(eq(entityConnections.entityAId, id), eq(entityConnections.entityBId, id))` in a count query.
**Warning signs:** Connections tab in ProfileTabs shows "0" badge while story view shows 12 connection cards.

## Open Questions

1. **Temporal clustering pattern (D-11, pattern 3)**
   - What we know: `entity_connections` has `firstSeen` and `lastSeen` dates; lobbying and vendor connections have both
   - What's unclear: A "close" overlap is undefined — 30 days? 90 days? The query logic for date range intersection across two different connection types for the same entity needs a concrete threshold
   - Recommendation: Default to 90-day window (lobbying registration within 90 days before contract award). Define as a named constant `TEMPORAL_CLUSTER_DAYS = 90` in the server function so it's easy to tune. If the SQL is complex (requires self-join across connection types), this pattern may need raw SQL via `db.execute(sql`...`)` rather than Drizzle's fluent API.

2. **Employer-sharing pattern (D-11, pattern 1)**
   - What we know: D-11 describes "multiple donors to the same politician sharing an employer/organization" — but `entity_connections` does not store employer relationships between donors. Employer data, if any, would be in `rawData` JSONB on the donations table.
   - What's unclear: Can this pattern actually be detected from `entity_connections` alone, or does it require a JOIN back to raw donations + JSONB extraction?
   - Recommendation: Scope the employer-sharing pattern conservatively for Phase 4.2 — if it requires JSONB extraction that isn't already indexed, skip it and use a simpler proxy (e.g., "entity donated to 3+ different politicians"). Flag in the plan as the pattern most likely to be deferred if SQL complexity is high.

3. **`sourceRecordIds` link target for callout cards (D-12/STORY-06)**
   - What we know: `entity_connections.sourceRecordIds` is a `text[]` column holding raw table row IDs. The connections table tab in `ProfileTabs` doesn't currently show record-level links — it shows the aggregated connection.
   - What's unclear: How should callout cards "link to the source records" when the existing UI doesn't have a per-record-ID view? Options: (a) link to the entity profile of the connected entity, (b) link to the raw government source URL (in `rawData.source_url`), (c) link to the connections tab for that entity pair.
   - Recommendation: For MVP, link to the connected entity's profile page (`/entity/$connectedEntityId`). This is always available and requires no new endpoints. Note this in the plan as a known simplification.
